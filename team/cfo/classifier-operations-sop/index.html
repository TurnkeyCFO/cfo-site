Purpose

This SOP tells a bookkeeper or reviewer exactly how transaction classification runs today: which client is on which lane, how the daily automated loop feeds the Close Command Transactions tab, how to review and approve coded transactions there, and what to do for a client the automated loop does not yet cover. The lanes exist in rule files a bookkeeper would never read; this page is where they become usable operating instructions.

The one thing to internalize: classification is a draft, and a human stands behind every posted code. The automation proposes; the reviewer decides; nothing writes back to the client's books without a human decision.

Owner and roles

Person Role in classification
Hardik (bookkeeper) Primary reviewer of proposed codes on the Transactions tab; runs the manual lane where no automated lane covers a client
Vanessa (US Client Lead) QC reviewer on the Transactions tab; final sign-off feeds the BD6 QC gate
Ricky (Founder) Owns any change to lane assignment, the shadow-to-live posture, and registration of new scheduled runs

The classification store itself (ledger.db) is single-writer: only the ledger-engine process writes it, and every reviewer decision flows through the Close Command Transactions tab, never by hand-editing the store.

Cadence and triggers

  • Daily, 7:45 AM CST: the ledger-engine ingest run (TurnkeyLedgerIngest) pulls each token-working client's uncategorized QBO transactions from the last 60 days, classifies them, and pushes them to the Close Command Transactions tab as queued items in shadow mode.
  • Daily / on demand: Hardik and Vanessa work the Transactions tab, approving, editing, or asking the client about each queued item.
  • Every 10 minutes: the ledger sync (TurnkeyLedgerSync) carries Ramp-lane proposals up and pulls reviewer decisions back down, stamping the Ramp approval sheet so the existing Ramp apply cron executes approvals.
  • Weekly (staged, not yet registered): the manual-lane run for any active client the automated loop does not cover. Registration is a Ricky decision (see The process).
  • Monthly close: classification review is a BD1 to BD6 activity feeding the QC gate; a client with a backlog of unresolved queued items is a BD3 escalation.

The process

The four lanes

Every onboarded client sits on exactly one primary classification lane. Never assume automation coverage; confirm a client's lane against this table or their dossier.

Lane What it is How review happens
Ramp classifier Ramp card transactions are classified automatically and pushed to the Transactions tab. On approval, the decision stamps the Ramp approval sheet and the Ramp apply cron writes it back Reviewer works the client's rows on the Transactions tab; write-back is via the existing Ramp apply path, never direct
ledger-engine QBO uncategorized (Transactions tab) The daily ingest pulls QBO transactions coded to an Uncategorized / Ask My Accountant account, classifies them in shadow mode, and queues them on the Transactions tab Reviewer approves or edits on the tab; QBO write-back is applied deliberately by the single-transaction applier, not on a cron
qbo-classifier pilot The older daily QBO classifier for the original pilot clients, still in shadow posture Slack-gated review; being superseded by the ledger-engine loop
None (manual) No automated lane covers the client; the bookkeeper categorizes every transaction in QBO directly Manual coding, reviewed under the normal QC pyramid

The daily loop, end to end

  1. 7:45 AM: the ingest run fetches uncategorized QBO transactions per token-working client, loads that client's reference context (its chart of accounts and classes, from the REF tab or a live pull), and classifies each line.
  2. Proposed codes that are not a valid account or class for that client are hard-rejected and flagged, so a bad suggestion never becomes a silent miscode. The transaction still queues, with live account and class dropdowns for the reviewer.
  3. Items push to the Close Command Transactions tab as queued, tagged shadow mode.
  4. Reviewers work the queue during the day (see below).
  5. Decisions are pulled back: Ramp-lane approvals stamp the Ramp sheet for the apply cron; QBO-lane approvals accumulate for the deliberate single-transaction applier. No lane writes to a client's books without a human decision.

Reviewer workflow on the Transactions tab

The Transactions tab lives in the Close Command portal (Ricky, Hardik, and Vanessa portals; not Eldine's). Open it, filter to your client, and work each queued row.

The staged weekly manual lane

For any active client the automated loop does not cover (today: a client whose QBO lane is eligible but not populating, or one deliberately on the manual lane), there is a staged weekly manual reclassification run so no client goes a full month without a pass.

  • What it is: a scheduled weekly task that reminds the bookkeeper to run a manual reclassification pass on the named uncovered clients: open each client's QBO, work the Uncategorized and Ask My Accountant accounts down, apply the client's coding rules from the dossier, and record the pass.
  • Task description (ready to register): "Weekly manual reclassification, uncovered clients. For each client on the manual lane, work Uncategorized and Ask My Accountant to zero or to flagged open items, apply dossier coding rules, and note completion. Owner: Hardik. Cadence: weekly."
  • Registration is a Ricky decision. The task configuration and schedule are prepared, but the scheduled job is not registered. Turning it on (or folding these clients into the automated loop instead) is Ricky's call, because it commits a standing weekly obligation and may overlap with the ledger-engine rollout. Until it is registered, uncovered clients are handled by the bookkeeper on the normal manual lane and flagged in the open-items list.

Guardrails

  • Never hand-edit the classification store. Every decision flows through the Transactions tab.
  • Never treat a proposed code as posted. Classification is shadow until a human approves and the appropriate apply path writes it back.
  • Never assume a client is covered. Confirm the lane and confirm the client is actually populating the queue before relying on automation for them.
  • Never free-type an account or class on the tab. Only real chart-of-accounts values are valid; an edit that is not in the client's synced lists is refused by design.
  • Never write back into a closed period. The QBO applier refuses closed-period writes; do not work around it.
  • Never register the weekly manual-lane job, change a lane assignment, or flip a client from shadow to live write-back without Ricky's decision.
  • A client producing no queue is classification unchanged for that client, not a client that is clean. Verify, then decide manual lane or fix the token.

Escalation

Situation What you do
A client shows no queued items and you cannot tell if it is clean or broken Check the daily run report; if the token is dead or the client is eligible but empty with known uncategorized activity, flag to Ricky
An active client is on no lane and is not being reclassified Handle manually this cycle, flag it on the open-items list, and raise registering the weekly manual lane with Ricky
Proposed codes are repeatedly wrong for a client Correct them, then flag the pattern with examples so the reference context improves
A queued item needs client input Use Ask Client on the tab; do not guess
You are unsure whether a transaction touches a closed period Do not apply it; flag to Ricky or the QC reviewer
Any change to lanes, schedules, or shadow-to-live posture Ricky, always
